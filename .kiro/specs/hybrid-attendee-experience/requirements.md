# Requirements Document

## Introduction

Hybrid events in Eventyay serve two distinct audiences — virtual attendees who participate online and in-person attendees who attend physically. Currently the platform treats all attendees identically regardless of how they registered, creating friction for virtual attendees who must navigate several clicks to reach the video/stream experience after checkout.

This feature introduces a `participation_mode` field on the `Product` model to classify ticket types, a centralized experience resolver service that derives each attendee's context from their purchased ticket, and a set of UX improvements that use that context to personalise routing, navigation, iCal exports, schedule display, and organizer email tooling. The design is intentionally extensible so future audience types (post-event viewers, speakers, interpreters, etc.) can be added without structural changes.

## Glossary

- **Product**: A ticket type in Eventyay (Django model `eventyay.base.models.product.Product`). Previously called "Item".
- **OrderPosition**: A single purchased ticket within an Order (`eventyay.base.models.orders.OrderPosition`).
- **ParticipationMode**: An enumeration of audience types attached to a Product. Initial values: `virtual`, `in_person`. Default: `in_person`.
- **ExperienceProfile**: The resolved set of behavioural attributes (stream access, primary CTA, calendar location, navigation visibility) computed for a specific attendee from their OrderPosition.
- **ExperienceResolver**: A service layer class (`eventyay.base.services.experience_resolver.ExperienceResolver`) responsible for computing an ExperienceProfile from an OrderPosition.
- **AttendeeOverride**: An optional per-OrderPosition field that allows an attendee (or organizer) to override the product-level ParticipationMode.
- **Stream Page**: The video/live-stream landing page reachable via the presale namespace.
- **Sendmail Plugin**: The `eventyay.plugins.sendmail` plugin used by organizers to send targeted emails.
- **iCal Export**: The `.ics` calendar file generated by `eventyay.presale.ical`.

## Requirements

### Requirement 1: Participation Mode on Product

**User Story:** As an event organizer, I want to assign a participation mode to each ticket type, so that the platform knows whether a ticket grants virtual or in-person access.

#### Acceptance Criteria

1. THE `Product` model SHALL expose a `participation_mode` field with choices `virtual` and `in_person`, defaulting to `in_person`.
2. WHEN an organizer saves a Product in the control panel, THE `Product` SHALL persist the selected `participation_mode` value.
3. THE control panel product form SHALL display the `participation_mode` field with a human-readable label and help text.
4. WHEN a Product is copied or duplicated, THE new Product SHALL inherit the `participation_mode` of the source Product.
5. THE `Product` model SHALL be extensible so that additional `participation_mode` values can be added in future without requiring a schema migration that drops existing data.

---

### Requirement 2: Attendee-Level Participation Mode Override

**User Story:** As an attendee or organizer, I want to override the participation mode on a specific ticket position, so that edge cases (e.g. a virtual ticket holder who will attend in person) can be handled without changing the product definition.

#### Acceptance Criteria

1. THE `OrderPosition` model SHALL expose an optional `participation_mode_override` field that, when set, takes precedence over the product-level `participation_mode`.
2. WHEN `participation_mode_override` is `None`, THE `ExperienceResolver` SHALL derive the participation mode from the associated `Product.participation_mode`.
3. WHEN `participation_mode_override` is set, THE `ExperienceResolver` SHALL use the override value instead of the product-level value.
4. IF an organizer sets `participation_mode_override` on an OrderPosition, THEN THE system SHALL log the change as a `LogEntry` on the Order.

---

### Requirement 3: Experience Resolver Service

**User Story:** As a developer, I want a single authoritative service that computes an attendee's experience context, so that participation-mode-dependent logic is not scattered across views, templates, and exporters.

#### Acceptance Criteria

1. THE `ExperienceResolver` SHALL accept an `OrderPosition` and return an `ExperienceProfile` data object.
2. THE `ExperienceProfile` SHALL contain at minimum: `participation_mode`, `has_stream_access` (bool), `primary_cta_url` (str or None), `calendar_location` (str or None), `show_join_online_nav` (bool).
3. WHEN `participation_mode` resolves to `virtual`, THE `ExperienceProfile` SHALL set `has_stream_access` to `True` and `show_join_online_nav` to `True`.
4. WHEN `participation_mode` resolves to `in_person`, THE `ExperienceProfile` SHALL set `has_stream_access` to `False` and `show_join_online_nav` to `False`.
5. THE `ExperienceResolver` SHALL be callable without a Django request object so it can be used from Celery tasks, exporters, and management commands.
6. WHEN the `OrderPosition` has no associated `Product`, THE `ExperienceResolver` SHALL raise a `ValueError` with a descriptive message.

---

### Requirement 4: Post-Checkout Routing

**User Story:** As a virtual attendee, I want to be taken directly to the stream/online access page after completing checkout, so that I can start watching the event without navigating through the general event page.

#### Acceptance Criteria

1. WHEN a checkout completes and all OrderPositions in the Order have `participation_mode` resolving to `virtual`, THE system SHALL redirect the attendee to the stream page for the event.
2. WHEN a checkout completes and at least one OrderPosition resolves to `in_person`, THE system SHALL redirect the attendee to the standard order confirmation page.
3. WHEN the event has no stream page configured, THE system SHALL redirect all attendees to the standard order confirmation page regardless of participation mode.
4. IF the stream page URL cannot be resolved, THEN THE system SHALL fall back to the standard order confirmation page and log a warning.

---

### Requirement 5: Navigation Personalisation

**User Story:** As a virtual attendee, I want to see a "Join online" navigation link on the event page, so that I can quickly access the stream without searching for it.

#### Acceptance Criteria

1. WHEN an authenticated attendee's active OrderPosition resolves to `virtual`, THE presale event navigation SHALL display a "Join online" link pointing to the stream page.
2. WHEN an authenticated attendee's active OrderPosition resolves to `in_person`, THE presale event navigation SHALL NOT display the "Join online" link.
3. WHEN no authenticated attendee session is present, THE presale event navigation SHALL NOT display the "Join online" link.
4. WHILE the event's stream page is not configured, THE "Join online" link SHALL NOT be displayed regardless of participation mode.

---

### Requirement 6: iCal Export Personalisation

**User Story:** As a virtual attendee, I want my calendar invite to contain the stream join URL as the location, so that I can join the event directly from my calendar application.

#### Acceptance Criteria

1. WHEN an iCal export is generated for an OrderPosition with `participation_mode` resolving to `virtual`, THE `ExperienceResolver` SHALL provide the stream join URL as `calendar_location`.
2. WHEN an iCal export is generated for an OrderPosition with `participation_mode` resolving to `in_person`, THE `ExperienceResolver` SHALL provide the physical venue address as `calendar_location`.
3. WHEN the `get_ical` function generates a `VEVENT`, THE system SHALL use the `calendar_location` from the `ExperienceProfile` as the iCal `LOCATION` field when an OrderPosition context is available.
4. IF no stream URL is available for a virtual attendee, THEN THE iCal `LOCATION` field SHALL fall back to the event URL.
5. WHEN `get_ical` is called with a plain `Event` object (no OrderPosition context), THE existing behaviour SHALL be preserved unchanged.

---

### Requirement 7: Schedule and Session UI

**User Story:** As a virtual attendee viewing the schedule, I want streamed sessions to show a prominent "Join" call-to-action, so that I can easily enter the session stream.

#### Acceptance Criteria

1. WHEN a schedule session is marked as streamed and the viewing attendee's participation mode resolves to `virtual`, THE session card SHALL display a prominent "Join" CTA button linking to the session stream.
2. WHEN a schedule session is not streamed, or the attendee's participation mode resolves to `in_person`, THE session card SHALL display the room or physical location instead of a "Join" CTA.
3. WHEN no authenticated attendee session is present, THE session card SHALL display the room or physical location.
4. THE session card rendering SHALL NOT make additional database queries per session beyond what is already fetched for the schedule view.

---

### Requirement 8: Sendmail Plugin — Participation Mode Filter

**User Story:** As an event organizer, I want to filter email recipients by participation mode in the Sendmail plugin, so that I can send targeted reminders to virtual or in-person attendees separately.

#### Acceptance Criteria

1. THE Sendmail compose form SHALL include an optional `participation_mode` filter field with choices: `(all)`, `virtual`, `in_person`.
2. WHEN the organizer selects `virtual`, THE system SHALL restrict the recipient OrderPosition queryset to positions whose effective participation mode is `virtual`.
3. WHEN the organizer selects `in_person`, THE system SHALL restrict the recipient OrderPosition queryset to positions whose effective participation mode is `in_person`.
4. WHEN the organizer selects `(all)` or leaves the field blank, THE system SHALL apply no participation mode filter and include all matching positions.
5. THE participation mode filter SHALL compose correctly with all existing Sendmail filters (order status, product, subevent, checkin status, date range).

---

### Requirement 9: Extensibility

**User Story:** As a developer, I want the participation mode system to be designed for extension, so that future audience types can be added without breaking existing behaviour.

#### Acceptance Criteria

1. THE `ParticipationMode` choices SHALL be defined as a Django `TextChoices` class in a dedicated module (`eventyay.base.models.choices` or equivalent) so new values can be appended without altering existing rows.
2. THE `ExperienceResolver` SHALL use a dispatch pattern (e.g. a method per mode or a registry dict) so that new participation modes can register their own profile-building logic without modifying the resolver's core method.
3. WHEN an unknown `participation_mode` value is encountered by the `ExperienceResolver`, THE resolver SHALL raise a `ValueError` with a descriptive message rather than silently returning a default profile.
4. THE `ExperienceProfile` data class SHALL use typed fields so that adding new profile attributes does not break existing callers that do not reference those attributes.
